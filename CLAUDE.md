# Blog 写作规范

## 目标
- 整体博客美观
- 整体博客易读
- 根据主题内容生成的博客内容完整
- 整体不违和

---

## 博客类型

| 类型 | 说明 | 风格要求 |
|------|------|----------|
| **技术文章类** | 纯技术案例、技术尝试类 | 技术原样输出，稍加修饰，保持完整性 |
| **观点类** | 一句话或一段话作为观点 | 不修改观点，仅配图和精要解说，简洁 |
| **调研类** | 一个疑问或几个问题 | 以技术专家身份调研，精要回复和解释，简洁 |

---

## 博客生成流程（P0 强制）

**触发时机**：用户发送一段话要求生成博客时

**执行流程**：

```
用户输入内容
    ↓
【步骤 1】确认是否创建博客
    ↓
用户确认要创建博客
    ↓
【步骤 2】检查是否指定博客类型？
    ↓
是 → 按指定类型生成
    ↓
否 → 向用户确认类型
    ↓
用户确认类型 → 按确认类型生成
```

**步骤 1 确认话术模板**：

```
检测到您可能想要创建博客，请确认：
1. 是否要创建博客？（回复"是"继续，"否"取消）
```

**步骤 2 确认话术模板**：

```
请确认要生成哪种类型的博客：
1. 技术文章类 - 纯技术案例、技术尝试
2. 观点类 - 围绕观点配图和解说，简洁为主
3. 调研类 - 以技术专家身份调研并精要回复
```

**硬规则**：
- ❌ 禁止 AI 自行判断博客类型
- ❌ 禁止 AI 自行决定是否创建博客
- ✅ 用户未明确确认"是否创建"前，禁止生成博客
- ✅ 用户未明确选择"类型"前，禁止生成博客

## 视觉美化标准

### 1. 表格格式

✅ 使用 Markdown 原生表格（不用 ASCII 表格）

```markdown
| 维度 | 评分 | 说明 |
|------|------|------|
| 工具熟练度 | ★★★★★ | 713 条会话，深度定制 |
```

✅ 多列对比使用表格

```markdown
| 以前 | 现在 |
|------|------|
| "帮我写个贪吃蛇" | "这个 Java 服务的性能瓶颈在哪里" |
| 代码能跑就行 | 有 lint 有测试有文档 |
```

### 2. 章节标题

✅ 每个二级标题添加相关 emoji 图标

```markdown
## 🎮 第一阶段：玩具期
## 💡 第二阶段：觉醒时刻
## 🏗️ 第三阶段：体系化
## 🔍 核心洞察
## 📈 使用模式的演变
## ✂️ 下一阶段：精简
## 📝 给你的建议
## 🌅 尾声
```

### 3. 分隔线

✅ 使用 `---` 分隔主要章节

```markdown
---

## 章节标题
```

### 4. 引用块

✅ 重要观点使用引用块 + 加粗

```markdown
> **证据 > 速度**

> **技术驱动型 | 配置爱好者 | 深度使用者**

> **因为最好的系统，不是不能再加东西，而是不能再减东西。**
```

### 5. 列表格式

✅ 普通列表（3 个以内）

```markdown
> - "帮你写个贪吃蛇"
> - "这个按钮怎么居中？"
> - "解释一下这段代码"
```

✅ 带状态标记的列表

```markdown
- ✅ **完善的配置体系**
- ⚠️ **凭证管理需要优化**
- ⚠️ **配置可以精简**
```

### 6. 强调格式

✅ 使用加粗强调关键词

```markdown
**能跑就行，不问出处。**

**证据 > 速度**
```

✅ 使用代码内联标记

```markdown
`CLAUDE.md`、`skills/`、`.env`
```

## 各类型博客规范

### 技术文章类

**适用场景**：
- 技术案例分享
- 技术方案尝试
- 工具使用经验
- 问题排查记录

**写作要求**：

- 技术原样输出，不过度修饰
- 保持内容完整性
- 可适当增加轻松风格的描述
- 禁止"事情是这样的"这类叙述

**模板结构**：
```markdown
---
layout: post
title: "标题"
date: YYYY-MM-DD
categories: blog
tags: [技术，标签 2]
author: nobt854
---

[简短开场，对话感]

[核心技术内容]

---

## [emoji] 章节标题

[内容]

---

## [emoji] 章节标题

[内容]
```

---

### 观点类

**适用场景**：
- 用户输入一句话作为观点
- 用户输入一段话作为观点

**写作要求**：

| 要求 | 说明 |
|------|------|
| 观点原样输出 | 用户的核心观点必须原文呈现，不做改写或扩展 |
| 生成配图 | 根据观点内容生成一张图文配合的插图（使用 SVG 或 ASCII 艺术形式） |
| AI 点评 | 基于用户观点进行精要点评，点到为止，不喧宾夺主 |
| 简洁为主 | 整体简洁，不长篇大论，不使用丰富的标题或内容 |

**观点类博客模板**：

```markdown
---
layout: post
title: "标题"
date: YYYY-MM-DD
categories: blog
tags: [观点，标签 2]
author: nobt854
---

## 💭 核心观点

> [用户观点原文，仅仅针对语法和标点符号等对原文进行必要润色，使用引用块呈现]

## 🎨 图解

[SVG/ASCII 配图，视觉化呈现观点]

## 🤖 AI 点评

[2-3 段精要点评，每段不超过 3 句话]

---

## 📌 延伸思考

[可选：1-2 个相关问题供读者思考]
```

**AI 点评写作要点**：
- 点到为止：不超过 3 段，每段 2-3 句话
- 视角补充：从技术/行业/人性等角度补充视角
- 不做评判：呈现思考，不做强弱判断
- 留有余地：用"可能"、"或许"等开放性词语

---

### 调研类

**适用场景**：
- 用户提出一个疑问
- 用户提出几个相关问题

**写作要求**：

| 要求 | 说明 |
|------|------|
| 专家身份 | 以对应领域技术专家身份调研并整理 |
| 精要回复 | 回复精要，不冗长 |
| 简洁解释 | 解释简洁，不长篇大论 |
| 不丰富标题 | 不使用丰富的标题或内容结构 |

**调研类博客模板**：

```markdown
---
layout: post
title: "标题"
date: YYYY-MM-DD
categories: blog
tags: [分析，调研，标签 2]
author: nobt854
---

[直接回复问题，开门见山]

## 🔍 核心分析

[精要分析内容]

## 📋 结论

[简洁结论]
```

## 格式约束规则

### 1. 表格强制转换

**规则**：发布前检查同类内容，如多行同类型数据未转为表格样式，必须转换

✅ 正确示例：
```markdown
| 占用率 | 状态 | 行动 |
|--------|------|------|
| <30% | ✅ 健康 | 继续 |
| 30-60% | ⚠️ 关注 | 考虑整理 |
| >60% | 🔴 警告 | 准备压缩 |
```

❌ 错误示例（纯文本列表）：
```
- <30%：健康，继续
- 30-60%：关注，考虑整理
- >60%：警告，准备压缩
```

### 2. 树形结构使用代码片段

**规则**：输入内容包含树形结构/流程图/步骤链时，必须使用代码片段包裹

✅ 正确示例：
```markdown

```
开始任务 → 每 30-45 分钟 → 人工整理 → 继续推进
```

```

❌ 错误示例（无包裹）：
开始任务 → 每 30-45 分钟 → 人工整理 → 继续推进

## 发布前强制验证（P0 级）

**触发时机**：创建或修改博客文件后，git commit 前

**验证步骤**：

1. **确认博客类型**
   - 检查是否已确认博客类型
   - 未确认 → 先向用户确认

2. **自动检查表格格式**
   - 检查所有表格是否有正确的分隔线 `|---|---|`
   - 检查表格分隔线后是否有内容行
   - 发现问题立即修复

3. **运行检查清单**
   ```bash
   # 检查博客文件 Markdown 语法
   - [ ] 已确认博客类型（技术/观点/调研）
   - [ ] 表格使用 Markdown 原生格式（非 ASCII 表格）
   - [ ] 章节标题有 emoji 图标
   - [ ] 主要章节间有分隔线 `---`
   - [ ] 重要观点使用引用块 + 加粗
   - [ ] 对比内容使用表格展示
   - [ ] 树形结构/命令示例使用代码片段包裹
   ```

4. **验证通过后再提交**
   - 检查不通过 → 修复 → 重新检查
   - 检查通过 → git add → git commit

**硬规则**：
- 未确认类型 → 禁止生成博客
- 未经验证的博客文件 → 禁止提交
