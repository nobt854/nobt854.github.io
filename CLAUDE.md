# Blog 写作规范

## 目标
- 整体博客美观
- 整体博客易读
- 根据主题内容生成的博客内容完整
- 整体不违和

## 视觉美化标准

### 1. 表格格式

✅ 使用 Markdown 原生表格（不用 ASCII 表格）

```markdown
| 维度 | 评分 | 说明 |
|------|------|------|
| 工具熟练度 | ★★★★★ | 713 条会话，深度定制 |
```

✅ 多列对比使用表格

```markdown
| 以前 | 现在 |
|------|------|
| "帮我写个贪吃蛇" | "这个 Java 服务的性能瓶颈在哪里" |
| 代码能跑就行 | 有 lint 有测试有文档 |
```

### 2. 章节标题

✅ 每个二级标题添加相关 emoji 图标

```markdown
## 🎮 第一阶段：玩具期
## 💡 第二阶段：觉醒时刻
## 🏗️ 第三阶段：体系化
## 🔍 核心洞察
## 📈 使用模式的演变
## ✂️ 下一阶段：精简
## 📝 给你的建议
## 🌅 尾声
```

### 3. 分隔线

✅ 使用 `---` 分隔主要章节

```markdown
---

## 章节标题
```

### 4. 引用块

✅ 重要观点使用引用块 + 加粗

```markdown
> **证据 > 速度**

> **技术驱动型 | 配置爱好者 | 深度使用者**

> **因为最好的系统，不是不能再加东西，而是不能再减东西。**
```

### 5. 列表格式

✅ 普通列表（3 个以内）

```markdown
> - "帮你写个贪吃蛇"
> - "这个按钮怎么居中？"
> - "解释一下这段代码"
```

✅ 带状态标记的列表

```markdown
- ✅ **完善的配置体系**
- ⚠️ **凭证管理需要优化**
- ⚠️ **配置可以精简**
```

### 6. 叙事风格
支持多种叙事风格的切换，默认采用技术原样输出风

**默认风格（根据主题/内容扩展）**：
- 技术原样输出风：根据输入的主题内容仅稍加修饰，大部分保持原样，增加全文博客的完整性
- 轻松幽默：在输入的主题内容基础上稍加扩展一些轻松风格的描述或元素，主调还是输入主题内容
- 你根据主要内容自行判断是否建议我新增和采用其它风格？
- 不要有类似这种描述：事情是这样的。

**观点类风格（用户明确指定时触发）**：

当用户输入一个观点并要求生成观点类博客时：

| 要求 | 说明 |
|------|------|
| 观点原样输出 | 用户的核心观点必须原文呈现，不做改写或扩展 |
| 生成配图 | 根据观点内容生成一张图文配合的插图（使用 SVG 或 ASCII 艺术形式） |
| AI 点评 | 基于用户观点进行精要点评，点到为止，不喧宾夺主 |

**观点类博客模板**：

```markdown
---
layout: post
title: "[观点] 标题"
date: YYYY-MM-DD
categories: blog
tags: [观点，标签 2]
author: nobt854
---

## 💭 核心观点

> [用户观点原文，使用引用块呈现]

## 🎨 图解

[SVG/ASCII 配图，视觉化呈现观点]

## 🤖 AI 点评

[2-3 段精要点评，每段不超过 3 句话]

---

## 📌 延伸思考

[可选：1-2 个相关问题供读者思考]
```

**AI 点评写作要点**：
- 点到为止：不超过 3 段，每段 2-3 句话
- 视角补充：从技术/行业/人性等角度补充视角
- 不做评判：呈现思考，不做强弱判断
- 留有余地：用"可能"、"或许"等开放性词语

发布前确认：
- [ ] 表格使用 Markdown 原生格式
- [ ] 章节标题有 emoji
- [ ] 主要章节间有分隔线
- [ ] 重要观点使用引用块 + 加粗
- [ ] 对比内容使用表格展示
- [ ] 语言风格有对话感
- [ ] 树形结构/命令示例使用代码片段包裹
- [ ] 观点类博客：观点原文呈现、配图视觉化、AI 点评精要（≤3 段）

## 格式约束规则

### 1. 表格强制转换

**规则**：发布前检查同类内容，如多行同类型数据未转为表格样式，必须转换

✅ 正确示例：
```markdown
| 占用率 | 状态 | 行动 |
|--------|------|------|
| <30% | ✅ 健康 | 继续 |
| 30-60% | ⚠️ 关注 | 考虑整理 |
| >60% | 🔴 警告 | 准备压缩 |
```

❌ 错误示例（纯文本列表）：
```
- <30%：健康，继续
- 30-60%：关注，考虑整理
- >60%：警告，准备压缩
```

### 2. 树形结构使用代码片段

**规则**：输入内容包含树形结构/流程图/步骤链时，必须使用代码片段包裹

✅ 正确示例：
```markdown
```
开始任务 → 每 30-45 分钟 → 人工整理 → 继续推进
```
```

❌ 错误示例（无包裹）：
开始任务 → 每 30-45 分钟 → 人工整理 → 继续推进

## 发布前强制验证（P0 级）

**触发时机**：创建或修改博客文件后，git commit 前

**验证步骤**：

1. **自动检查表格格式**
   - 检查所有表格是否有正确的分隔线 `|---|---|`
   - 检查表格分隔线后是否有内容行
   - 发现问题立即修复

2. **运行检查清单**
   ```bash
   # 检查博客文件Markdown 语法
   - [ ] 表格使用 Markdown 原生格式（非 ASCII 表格）
   - [ ] 章节标题有 emoji 图标
   - [ ] 主要章节间有分隔线 `---`
   - [ ] 重要观点使用引用块 + 加粗
   - [ ] 对比内容使用表格展示
   - [ ] 树形结构/命令示例使用代码片段包裹
   ```

3. **验证通过后再提交**
   - 检查不通过 → 修复 → 重新检查
   - 检查通过 → git add → git commit

**硬规则**：未经验证的博客文件禁止提交
